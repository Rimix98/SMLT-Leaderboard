## T1: Backend — Path Traversal в player history
**Файл**: `handler/handlers_players.go:388`
**Проблема**: `playerID` извлекается из URL без валидации — возможен path traversal (`../../../`)
**Решение**: Добавить валидацию `playerID` через regex `^[a-zA-Z0-9 _.\-]{1,64}$` перед запросом в Firestore

---

## T2: Backend — Unsafe type assertion в cleanup routines
**Файлы**: `handler/captcha.go:93`, `handler/auth.go:499`
**Проблема**: `value.(*int)` и `doc.DataTo` — panic если тип не совпадает
**Решение**: Использовать comma-ok assertion (`val.(*int)` → `if p, ok := val.(*int); ok {`)

---

## T3: Backend — Bounds check перед swap в reorder
**Файл**: `handler/handlers_staff.go:507-515`
**Проблема**: `data.Roles[idx]` без проверки `idx >= len(data.Roles)` → panic
**Решение**: Добавить bounds check для `idx` (не только `target`)

---

## T4: Backend — Context timeouts на Firestore queries
**Файл**: `handler/handlers_projects.go:32`, `handler/handlers_staff.go:35`, `handler/handlers_players.go:398`
**Проблема**: Нет `context.WithTimeout` — зависший Firestore запрос блокирует handler навсегда
**Решение**: Обернуть ctx в `context.WithTimeout(ctx, 10*time.Second)` для каждого Firestore запроса

---

## T5: Backend — Graceful degradation для внешнего API
**Файл**: `handler/handlers_players.go:200-273`
**Проблема**: Если Demonlist.org down — leaderboard handler зависает на worker pool без таймаута
**Решение**: Добавить context timeout на весь leaderboard fetch (30s), вернуть частичные данные если часть игроков не загрузилась
